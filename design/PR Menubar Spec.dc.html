PR Menubar — v1 spec

macOS menubar app (Electron) · July 3, 2026 · draft for review

Philosophy: ship lean, evolve through use. Everything here is v1-minimal; anything marked PROPOSAL is a default you can veto, and LATER is explicitly out of v1.

1. What it is

A popover from the macOS menubar (≤560px wide, native vibrancy, light + dark following system appearance) that answers one question: what should I do next on my PRs? PR lists + CI status only. No issues, no dependabot, no notifications feed.

2. Setup & data

  • Auth: reuses gh CLI credentials. If gh auth status fails, the popover shows a single setup screen with the command to run.
  • Repos: starts with 2, user-configurable list in Settings. No auto-discovery in v1.
  • Refresh: poll every 60s while popover is closed; refresh immediately on open; manual ⌘R. Footer shows "Synced Ns ago".

3. Tabs

  • All — every PR the app watches, one feed, sorted by next-action urgency.
  • My PRs — open PRs I authored in watched repos.
  • Reviewing — open PRs where my review is requested OR I've already reviewed/commented and the PR is still open.
  • Team — open PRs authored by a hand-picked set of GitHub usernames (not filtered by my involvement). Master list edited in Settings; a pill row at the bottom of the tab toggles individual people in/out on the fly.
  • Saved — any PR I've starred, from any tab. Manual bookmark, survives across tabs.

4. The list (per design 3a/3b)

Row: CI dot · title · repo #num · updated-ago · next-action chip · review-status pill · author avatar. Click row = expand; ⌘-click or double-click = open in browser.

Next-action chip (the core feature) — computed per PR, drives sort order within each tab:

ChipCondition (first match wins)
FIX CImy PR, a meaningful check failed (see §6)
MERGEmy PR, approved, checks green, no conflicts
ADDRESSmy PR: changes requested, unresolved review threads, or merge conflicts
REVIEWreview requested from me, I haven't started
RESUMEI have a pending/partial review
WAITINGeverything else (waiting on reviewers, CI running, draft)

How RESUME is detected — GitHub stores started-but-unsubmitted reviews server-side (API review state PENDING). RESUME = I have a pending review draft, OR I commented without submitting a review, OR my last review predates the newest commit (re-review).

Expanded row: per-check CI list (name, status, duration, "View log" on failures), "Re-run failed", and the action strip: Open ↗ · copy branch name · star · snooze.

Grouping PROPOSAL: flat list sorted by chip urgency on My PRs / People / Saved; grouped headers (Start review / Continue / Waiting) on Reviewing only, where the groups are natural.

5. Row actions

  • Open in browser — PR page; "View log" opens the failing check directly.
  • Copy branch name — one click, toast "Copied".
  • Star — adds to Saved.
  • Snooze — small menu: 1 hour · until tomorrow · until activity. "Activity" = new commit, new review, new comment, or CI state change on that PR. Snoozed rows hide from all tabs (except Saved) and return automatically; a footer count ("2 snoozed") un-hides them.
  • Done PROPOSAL — same as snooze-until-activity, just named for the "I've dealt with this" gesture. One mechanism, two entry points.
  • Merge PROPOSAL — in-popover merge with one confirm step, only shown on MERGE-chip rows. Trivial to remove if it feels dangerous; "open GitHub to merge" is always available.

6. CI status & the codecov problem

Raw GitHub check status is too noisy to act on. v1 ships a noisy-checks list (Settings, per-repo or global, e.g. codecov/*):

  • A noisy check's failure never triggers a FIX CI chip, red dot, or notification while other checks are still running.
  • Once all non-noisy checks complete: if they're green, the PR is treated as green — a still-failing noisy check is shown in the expanded breakdown (marked "ignored") but doesn't change the dot. If a noisy check is the only failure at completion, the row shows an amber dot, not red.
  • Non-noisy failures report immediately (a real build/test failure is actionable even mid-run).

Dot colors: green = all meaningful checks pass · red = meaningful failure · amber = running, or only-noisy failure · hollow amber = queued · gray = no checks.

7. Notifications (macOS)

  • CI fails on my PR — meaningful failures only, per §6, and only on transitions (green/running → failed), never repeats.
  • My PR becomes approved / mergeable.
  • Someone requests my review.
  • New comments on my PRs — batched: at most one notification per PR per 15 min ("3 new comments on #482").

Clicking a notification opens the PR in the browser. All four are individually toggleable in Settings.

8. Menubar icon PROPOSAL

Icon + a small count of PRs where the next action is mine (FIX CI + MERGE + ADDRESS + REVIEW + RESUME on My PRs/Reviewing only — never the People tab, so watching dozens of PRs doesn't pollute it). No count when zero. Not a "broken things" indicator — it's a to-do count, which stays small by construction. Toggleable to plain icon in Settings.

9. Settings (one small window)

Repos to watch · People list (for People tab) · noisy-checks patterns · notification toggles · badge on/off · launch at login.

10. Explicitly not in v1 LATER

Triage/focus mode (revisit after real use) · issues, dependabot · multiple GitHub accounts · in-app review/commenting · keyboard-shortcut layer beyond ⌘R · auto-repo discovery · team/org syncing.

11. Open questions
